# Project agent memory

This file is the project's committed home for project-intrinsic agent knowledge: build, test, release, architecture, and sharp-edge notes that should travel with the code.

- Scenario content (quiz questions + rink animations) lives in `src/data/scenarios/*.ts`; the authoring contract, including puck placement rules, is `src/data/scenarios/AUTHORING.md`. Mechanical invariants are enforced by `npm run validate` (also part of `npm run build`). `freezeLine` goes straight into the situation/question - no "Stop it here"/"Freeze"/"Hold it" transition opener (the whistle + freeze frame already stop the tape).
- The rink renderer draws a carried puck on the carrier's stick: `puckOnStick` in `src/components/RinkDiagram.tsx` shifts any puck within 3.6 units of a player to that player's rim. Author carried pucks at the carrier's exact coordinates and let the renderer handle the offset.
- Real Examples: real moments from the player's own games, seeded in `src/data/scenarios/real-examples.ts`. A scenario carries the optional `realGame` field (`RealGameExample` in `src/types/index.ts`) - its presence is the only marker. It routes the scenario into the hockey home screen's "Real Examples" section (`startSession('real')`, a `SessionMode`) and makes `SessionScreen`/`FeedbackScreen` frame it as "this happened in your game" (banner + `YOUR GAME` pip). `realGame` scenarios are excluded from every generic pool (Daily 5 / Boss / Weakest / zone tracks all filter `!s.realGame`; `weakestCategory` and the `accuracyForDifficulty` unlock gate skip them too), so the section is opt-in, never gated by unlocks, and can't nudge the unlock gate. Adding a future example is a single `Scenario` entry with `realGame` set - no UI work, but it still needs a full animation + `npm run narrate` like any scenario. The section auto-hides when a module has none (so lacrosse never shows it). Coaching-content rule (safe play): a hockey scenario's taught/correct answer must never be a pass across the middle of your OWN end; offensive-zone scoring passes and neutral-zone north-south seams are approved exceptions - see the "Safe-play rule" in `src/data/scenarios/AUTHORING.md`.
- Shared UI must speak the active sport: the pedagogical engine is sport-agnostic (see `src/data/modules.ts`), so chrome both modules render - `Onboard`, `CoachMode` - takes a `sport` prop threaded from `activeModule.sport` in `App.tsx` and words its copy per sport; Coach Mode's JSON export also filters `SCENARIOS` to the active sport (`sportOf`). A lacrosse user must never read hockey terms (rink/ice/puck/boards) - thread the sport through any new shared surface instead of hardcoding hockey wording.
- Voice narration is one coach voice (ElevenLabs), everything spoken, no browser SpeechSynthesis (never reintroduce it). Every scenario is animated and fully voiced, so every mode - including Boss, which draws a rookie-excluded, elite-leaning subset of the bank via `pickBossScenarios` - has audio. `npm run narrate` (`scripts/generate-narration.ts`) renders per-scenario clips (`scenarioClips` in `scripts/narration-manifest.ts`): the animation voice-over (key `<id>`), the freeze-line prompt (`<id>.freeze`), each read option (`<id>.opt.<i>`, text via `optionSpeech`), and the results-beat feedback (`<id>.fb` = correct answer stated + its rationale, via `feedbackSpeech`; for tap the label is spoken verbatim). Plus six scenario-independent openers (`STATIC_OPENER_CLIPS`, keys `fb.correct.<i>`/`fb.wrong.<i>`) threaded into generation/`expectedManifest` via the `extraClips` param. Files are `public/audio/<key>.<hash>.mp3`; the MP3s + `public/audio/manifest.json` are committed build artifacts, so the app/CI never need a key (`ELEVENLABS_API_KEY` in `.env` only for regeneration; see `.env.example`). Runs are skip-existing/content-hashed and prune-safe without a key. Sharp edge: `contentHash` joins fields with a NUL byte (`\x00`), not a space, and folds in the ElevenLabs `voiceSettings` (stability/similarity/style/speaker-boost and optional `speed`) alongside voice/model/format/text - retuning the delivery correctly re-renders every clip, but changing the separator or the settings-serialization forces a full (billable) re-render. Current tune (in `generate-narration.ts`) is a steady, warm coach at natural pace: `voiceId UgBBYS2sOqTuMpoF3BR0` (captain-selected; override via `ELEVENLABS_VOICE_ID`), `stability 0.5, style 0.28, similarityBoost 0.85, speaker-boost on, speed 1.0`; `speed` is optional and appended to the hash only when set, so older no-`speed` configs/tests keep their historical hash. The voice was swapped once (an earlier warmer/slower take was rejected on ear-check) - expect voice/pace to be captain-tuned, so treat these values as current-not-sacred. The client (`src/lib/narrationAudio.ts`) only loads static files - never the API: `AnimatedRink` plays `<id>` during the animation; `SessionScreen` awaits `<id>.freeze` then, in shuffled display order, each option's clip by its ORIGINAL data index (`<id>.opt.<origIdx>`) during the reveal (options are shuffled per question - `displayOptions` carries `origIdx` - to kill the always-B exploit, and clips are keyed to `origIdx` so the coach reads the text the highlighted button shows; sharp edge: `optionSpeech` bakes the spoken letter A/B/C and the "Do you,"/"Or," lead into each clip by original position, so after a shuffle the spoken letter/framing can lag the on-screen letter - only fully fixable by re-rendering position-independent clips, which needs the key); `FeedbackScreen` plays a rotating opener (`nextFeedbackOpener`) then `<id>.fb` on the results beat for both right and wrong answers; `stop()` cuts audio on sound-off/skip/replay/quit/unmount. Autoplay unlock (critical): clips start from timers/effects, never synchronously inside a click, so `narrationAudio` keeps a small pool of `<audio>` elements primed once inside the first user gesture (`unlock()`, wired to pointer/touch/key in `App.tsx`) and ping-pongs/reuses them - a bare `new Audio().play()` inside a promise `.then` is blocked by iOS/Safari's autoplay policy and silences ALL narration. Browser bindings (Audio/timers/fetch/base URL) are injected via `NarrationEnv` so the pool/fade/sequencing logic is DOM-free unit-tested by `scripts/narration-audio.test.ts` (asserts the fade taper never silences the incoming clip, that `unlock()` primes every pooled element, and the stale-manifest recovery paths described in the PWA entry below). Playback timing constants + the pure `fadeRamp` helper live in `src/lib/narrationTiming.ts` (unit-tested by `scripts/narration-timing.test.ts`): `stop()` fades a still-playing clip to silence over `CLIP_FADE_MS` before pausing (a hard pause mid-waveform clicks/garbles at section boundaries), and `SessionScreen`'s reveal inserts an `OPTION_LEAD_IN_MS` beat after the freeze prompt before option A plus an `OPTION_GAP_MS` beat between option reads so the coach isn't rushed. Missing audio = silent.
- PWA: `vite-plugin-pwa` (config in `vite.config.ts`) makes the app installable and offline-capable. It runs in **`injectManifest` mode** with a hand-written service worker at `src/sw.ts` (NOT `generateSW`); `npm run build` bundles it (workbox inlined, classic script - no `type:module`, so iOS-safe) into `dist/sw.js` + emits `dist/manifest.webmanifest`, and the plugin auto-injects the manifest link and `registerSW.js` into `index.html`. The app shell (js/css/html/icons/fonts) is precached; `/audio/*` clips (CacheFirst, cached lazily as they play) and Google Fonts (CacheFirst/SWR) are runtime-cached - all routes live in `src/sw.ts`. Sharp edge that forced the hand-written SW (was a real "no sound" bug): narration mp3s play via `new Audio()`, which always sends a `Range` header, so a range-supporting host (Vercel) answers `206 Partial Content` - and the Cache API **refuses to store a 206** (`cache.put` throws "Partial response ... unsupported"). Under the old `generateSW` `CacheFirst`+`rangeRequests` config every clip write silently failed, `audio-clips` stayed empty, and the PWA had NO offline audio (silent narration on the flaky/absent wifi typical of a rink). `src/sw.ts`'s audio route strips the `Range` header before the network fetch (`stripRangeHeaderPlugin`) so the SW caches a full `200`; the byte range the audio element asked for is then carved out of the full body on every path - by `RangeRequestsPlugin` on cache hits and by `partialContentPlugin` (`ensurePartialResponse` + Workbox's `createPartialResponse`) on the cache-miss network 200, because WebKit's media loader (iOS Safari, the primary installed platform) requires a genuine `206` answer to a range request. Keep the strip plugin + `RangeRequestsPlugin` + `partialContentPlugin` together, do NOT re-add `206` to that route's cacheable statuses, and leave non-200s unconverted (a stale-manifest 404 must stay an error so narration recovery fires). The clip cache's `ExpirationPlugin` bounds by `maxEntries` only - never add `maxAgeSeconds`: `CacheFirst` never refreshes a cached entry's timestamp, so an age cap silently expires any clip first cached beyond the window and re-silences a long-offline rink (rationale in `src/sw.ts`). The pure halves live in `src/lib/swRangeStrip.ts` (tested by `scripts/sw-range-strip.test.ts`); the seven `workbox-*` packages the SW imports are declared devDependencies - keep them version-matched to the workbox that `vite-plugin-pwa`/`workbox-build` resolves. `src/sw.ts` is type-checked separately (WebWorker lib) via `tsconfig.worker.json` and is excluded from the main `tsconfig.json`. Stale-shell handling (so returning users, esp. installed iOS PWAs, aren't stuck a deploy behind): `registerType: 'autoUpdate'` gives the SW `skipWaiting`+`clientsClaim`, and `App.tsx` reloads the page once on `controllerchange` so the new shell lands on the next visit - guarded by `wasControlled` (skips only the initial claim on a first install - a deploy activating later that same session still reloads) plus a one-shot flag (no loop). Deliberately a silent reload, not a "tap to refresh" prompt: kids won't act on a banner and durable progress lives in localStorage (only the in-flight run is React state). Stale-manifest recovery: the audio manifest maps scenario keys to content-hashed mp3 filenames that get pruned across deploys, so a cached manifest can 404 a clip (pruned filename) or lack a key entirely (clip added in a newer deploy) and silence a session; `narrationAudio.ts` refetches `manifest.json?fresh=1` (cache-bypassing) in either case and retries once with the fresh mapping. Keys still absent after a successful refetch are negative-cached so replays stay silent without refetching every play (cleared when a refetch adopts a changed manifest; a failed/empty refetch leaves the key retryable), and non-404 play failures (autoplay-blocked `play()`, a load error after `stop()`) resolve silently with no refetch. For that escape hatch to reach the network the SW manifest route matches the bare pathname only (`=== '/audio/manifest.json' && !url.search`) and the broad `/audio/` CacheFirst rule excludes the manifest - keep both if you touch the routing. Icons in `public/` (`favicon.svg`, `pwa-192x192.png`, `pwa-512x512.png`, `maskable-icon-512x512.png`, `apple-touch-icon.png`) are committed artifacts rasterized from `favicon.svg` by `scripts/generate-icons.mjs` - `sharp` is intentionally not a committed dep, so regenerate with the one-liner in that script's header. To verify installability/offline you must use a real prod build (`npm run build && npm run preview`), not `npm run dev` (SW is disabled in dev by default).
- `npm run lint` is `tsc --noEmit` over `src` (DOM lib, excludes `src/sw.ts` and `scripts/`) plus `tsc --noEmit -p tsconfig.worker.json` for the WebWorker-lib service worker (`src/sw.ts`); `npm run build` runs both too. `npm test` runs the `scripts/*.test.ts` unit tests via `node --test`; the narration generator's logic is tested with the API mocked. Visual/audio changes to animations still need eyeball/ear verification in the running app (`npm run dev`); PWA/offline-audio (service-worker) changes need a prod build + `npm run preview` since the SW is disabled in dev.
